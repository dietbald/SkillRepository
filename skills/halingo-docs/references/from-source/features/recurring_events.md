# Recurring events

> **Provenance:** code-derived from `/home/tj/Repos/Halingo-Main`. Helpdesk coverage: partial — see `../../full_documentation/calendar_appointments.md` (recurrence is mentioned for creation but the "edit series vs. one" mechanics are not). Verify against running app before promoting to `manual/`.

## What it is

Halingo models recurrence by **materialising every occurrence** as its own row in the `events` collection, then linking them together through a shared `repeatId`. There is no separate `recurrences` collection and no "virtual" occurrences expanded at read time — each session of a weekly series is a real document with its own invoice linkage, `hasPayBack` flag, and Rosa id.

## Where it lives in the UI

- **Create**: `EventAddBox` (`app/imports/modules/calendar/components/infobox/EventAddBox.jsx`) contains a `Switch` labelled `REPEAT` (line 602-608). Enabling it swipes the modal into a second tab showing `RepeatForm` (`app/imports/modules/forms/forms/RepeatForm.jsx`). The switch is hidden when `meta.subType === "GROUP"` (line 600 — group events cannot recur, see [group_events.md](group_events.md)).
- **Edit**: the top of `AppointmentEventPage`, `MeetingEventPage`, `PrivateEventPage`, and `ConsultationEventPage` all render an `EVENT_HAS_REPEAT_DESCRIPTION` toggle when `event.repeatId` is set (`AppointmentEventPage.jsx:346-356` and equivalents). When on, edits are applied to the whole series.
- **Remove**: `RemoveButtons.jsx:32-45` adds a second "Remove repeated" button for recurring events.

## Data model

### Fields on `events`

```js
// events.jsx:1290-1301
repeatId: { type: String, optional: true },
groupId:  { type: String, optional: true },
repeat:   { type: repeatDateSchema, optional: true },
```

- `repeatId` — shared string id on every occurrence of a series. Generated by `new Mongo.ObjectID()._str` in `_repeatEvent` (`server/util.jsx:433`).
- `repeat` — only present on the *seed* event (or during creation). It carries the recurrence pattern and is not copied to every child.

### The `repeatDateSchema`

`app/imports/lib/simpleSchemas/util.jsx:79-177`:

| Field | Type | Allowed values |
|---|---|---|
| `type` | Number | 1 = DAILY, 2 = WEEKLY, 3 = YEARLY (`repeatDateSchemaEnums.types`, lines 80-93) |
| `frequency` | Number (optional) | e.g. every 2 weeks → `2` |
| `weeklyOn` | `Number[]` (optional) | 0-6 (Sun-Sat). Only used for WEEKLY. |
| `startDate` | Date | When the series begins. |
| `endType` | Number | 1 = NUMBER_SESSIONS, 2 = END_DATE, 3 = END_DATE_PAYBACK (lines 94-107) |
| `endDate` | Date (optional) | Used when `endType === 2` or `3`. |
| `numberSessions` | Number (optional) | Used when `endType === 1`. |
| `treatment` | Object (blackbox, optional) | — |

## Methods (Meteor)

### Create — `_repeatEvent`
`app/imports/api/events/server/util.jsx:432-499`:

1. Generates `repeatId = new Mongo.ObjectID()._str`.
2. If the form used `endType: 3` ("end at bilan end"), it reads the currently valid bilan on the treatment, copies `bilan.end` as `endDate`, and sets `numberSessions = TreatmentsUtil.getSessionsLeft(treatment)` (lines 437-448). If there is no valid bilan, throws `"no_valid_bilan_found"`.
3. Walks the recurrence via `Util.repeatDate(event.repeat, cb)` (`app/imports/lib/util/util.jsx:120-175`). The walker:
   - For DAILY: adds `frequency * n` days per iteration.
   - For WEEKLY: for each weekday in `weeklyOn`, clones the start and jumps by `frequency` weeks.
   - For YEARLY: adds `frequency * n` years per iteration.
   - Stops when `count >= numberSessions` or `newDate.isAfter(endMoment)`, with a hard cap of 300 iterations (line 151).
4. For each generated date, clones the base event and sets `start`/`end` by shifting, preserving time of day via `Util.setDateKeepTime`.
5. For type-1 events with a treatment, computes `hasPayBack` per occurrence using `canBePaidBack(newEvent, false, treatment, allEvents)`, where `allEvents` is a rolling window that includes already-created occurrences. This is why occurrences early in the series may be reimbursable while later ones (past the bracket) are not.
6. Decrements `sessionsLeft` as it goes. If `endType === 3` and `sessionsLeft < 1`, the callback returns `true` to halt iteration — i.e. "stop at the first occurrence that would run out of reimbursable sessions".
7. Inserts everything with `Events.batchInsert(eventsToAdd)`.
8. Triggers `CommissionInvoicesUtil.updateCommissionNewEvents` and re-schedules the `TreatmentSessionObserver` notification.

### Update — scoped by repeatId
`events.update` at `methods.js:258-353`:
- When `repeatId` is provided in the call, the update target is `{ repeatId }`, which applies `{ multi: true }` (line 342) to all siblings.
- Only a whitelist of fields is allowed to propagate to the series:

  ```js
  // methods.js:282-285
  $set: repeatId
    ? _.pick(fields, ["title", "meta", "price", "kmCompensation", "color", "description"])
    : _.omit(fields, ["sessionCount"]),
  ```
  — i.e. time of day and patient/treatment cannot be changed for an entire series in one operation.
- Events that already have an `invoiceId` are excluded from the update via `query.invoiceId = null` (line 340).
- After the update, `RosaEvents.pushEventsToRosa(eventsToUpdate.map(e => e._id))` re-pushes every modified sibling.

### Remove — scoped by repeatId
`removeTheEvent(event, repeatId)` at `methods.js:44-77`:

```js
if (repeatId) {
  const events = Events.find({
    start: { $gte: event.start },   // only from this occurrence forward
    repeatId,
    invoiceId: { $exists: false },  // never delete invoiced events
  }).fetch();
  ...
  Events.remove({ _id: { $in: _.map(events, "_id") } }, { multi: true });
  await RosaEvents.pushEventsToRosa(_.map(events, "_id"));
}
```

Key details:
- "Remove repeated" deletes **this occurrence and every later occurrence in the series**, never earlier ones. To delete the whole series you must act on the first occurrence.
- Invoiced occurrences are silently skipped (the `Events.remove` override at `events.jsx:9-12` also guards this with `invoiceId: { $exists: false }`).
- The Rosa push batches deletions.

The UI confirmation dialog (`RemoveButtons.jsx:37-44`) shows the current occurrence date to make "from this date onward" clear.

If `hasRepeat` was false, the server takes the non-repeat branch in `_addEvent` (`server/util.jsx:501-514`).

### Edit — "apply to all repeated" toggle
Every per-type edit page checks `event.repeatId` and renders the `EVENT_HAS_REPEAT_DESCRIPTION` switch at the top (`AppointmentEventPage.jsx:346-356`). When toggled on, subsequent field changes pass `changesToAllRepeat && event.repeatId` as the `repeatId` argument to `events.update` — the server then applies the `repeatId` whitelist from above.

Because time of day is **not** in the whitelist, the start/end date pickers are `disabled={changesToAllRepeat}` in the `AppointmentEventPage` (lines 507, 518), `MeetingEventPage` (lines 111, 119), and `PrivateEventPage` (lines 113, 121). The user must turn "apply to all" off to re-time a single occurrence.

### Remove — two buttons
`RemoveButtons.jsx:24-46`:
- `REMOVE` — remove this single occurrence.
- `REMOVE_REPEATED` (only shown if `event.repeatId`) — open a confirm dialog "Remove all repeated events from {date}" then call `events.remove` with the series id.

## Permissions

- Edit/remove of a series respects the same `practice.events.add.otherUser` check as a single event.
- An invoiced occurrence cannot be removed even as part of a series operation — protected by `invoiceId: { $exists: false }` in both the `Events.remove` override and the explicit query in `removeTheEvent`.

## Notable details

### "End at end date of bilan" is powerful
`endType: 3` (`END_DATE_PAYBACK`) lets the therapist say "make weekly sessions until this patient's current bilan runs out or until the RIZIV bracket is exhausted, whichever comes first". The server does the math and may stop early. This is the only way in the UI to bind recurrence to the treatment lifecycle automatically.

### Iteration hard cap
`Util.repeatDate` aborts after 300 iterations (`util.jsx:151`). A daily series with no end type set would still be capped. In practice the create path always supplies either `numberSessions` or `endDate` so this is a defensive ceiling.

### No virtual occurrences
Every Friday appointment in a 52-week series is a separate row with its own `_id`, `hasPayBack`, `invoiceId`, etc. This means:
- Bulk actions scale linearly with occurrences.
- The Rosa push is a per-row operation (batched 50 at a time, `rosa-events.ts:301-307`).
- Changing the pattern after creation is impossible — you must delete the series and recreate. There is no "edit recurrence rule" UI.

### Rosa recurring push
`rosa-events.ts:474-557` has a separate code path (`pushRecurringEventsToRosaForUser`) that groups events by `repeatId` and posts them to Rosa as `RosaApi.events.createRecurringEventsForUser` batches. Rosa's own recurrence semantics are preserved there — it does understand "this group of events is a recurring series" via `recurrenceId: repeatId`.

### `repeat` subdocument vs. `repeatId`
`repeat` only exists on the create payload and, briefly, on the base event before fan-out. It is not copied to every sibling. `repeatId`, conversely, exists on every sibling and is the only way to locate the series after the fact.

### Invoiced occurrences shadow the series
A subtle consequence of the "skip invoiced" rule: if occurrences 1-5 are invoiced and the user clicks "remove repeated from occurrence 3 onward", occurrences 3, 4, 5 are silently untouched and 6+ are removed. The user gets no warning.

### Edit history
There is no versioning or audit of recurring-series edits. Rolling back an accidental "apply to all" is manual.

## Helpdesk overlap

The helpdesk describes the create-recurrence flow and shows screenshots of the form. It does not explain:
- The "end at bilan end" option's actual stopping rule.
- That "edit applied to all" only propagates a whitelist of fields.
- That "remove repeated" only removes forward from the current occurrence.
- That invoiced occurrences are silently skipped on bulk operations.
