## What it is

A clinical report ("verslag") in Halingo is one of three things stored under the dossier's *Documenten* tab:

1. A **rich-text report** authored inline in Halingo via a `TextEditorLoadable` (`PatientFileReportPage.jsx`). Stored in the `patientFileReports` mongo collection.
2. An **uploaded document** (PDF, image, audio, etc.) stored in S3 via the `Documents` collection — see `patient_documents.md`.
3. A **generated demand form** (`aanvraagformulier voor terugbetaling` / `kennisgeving van verlenging`) — a fill-in PDF produced server-side from a stock template, then stored as a normal `Documents` row. Generated by `addPatientFileDemandForm` (`patientFileReports/methods.jsx:134-171`) which calls `PatientFilesUtil.generateDemandForm` on the server (`patientFiles/server/util.js:34-106`).

This document describes types 1 and 3 — type 2 is in `patient_documents.md`.

## Where it lives in the UI

- **Dossier tab** — "Documenten" (i18n key `patient.reports.title`, `client/nl.i18n.js:446 → "Documenten"`). Tab index 3 in the standard tab list (`PatientFileTabs.jsx:96-108`). Renders `PatientFileReportsOverviewContainer`.
- **Overview page** — `PatientFileReportsOverviewPage.jsx`. Combines `patientFileReports` and `Documents` into a single search index (`PatientFileReportsOverviewPage.jsx:146`) using `PatientFileDocumentsIndex` and `PatientFileReportsIndex` from `lib/search-indexes/`. Filter chips: `audio` / `doc` (text reports + PDF + plain text) / `pic` (`PatientFileReportsOverviewPage.jsx:139-145`). Sort by name asc/desc or date asc/desc (`:157-162`).
- **Add popup** — `AddReportPopup.jsx`. A modal with three tabs:
  - **Aanmaken** — name + tags → creates an empty `patientFileReports` row.
  - **Uploaden** — file picker + tags → uploads to `Documents` (S3).
  - **Formulieren** — date + first/extension toggle + "fill in administrative info" switch + tags → server-generates a demand-form PDF (only when `enableDemandForm`).
- **Report editor page** — `app/imports/modules/patientfiles/reports/PatientFileReportPage.jsx`. Routes:
  - `/patients/:patientId/reports/:reportId` (top-level)
  - `/patients/:patientId/treatments/reports/:reportId` (under treatments tab)
  Both render the same component (`PatientFileReportContainer`) — the second route exists so reports created from a treatment open in a context that knows the originating treatment id.

## Data model

Collection: `patientFileReports` · `app/imports/api/patientFileReports/patientFileReports.jsx:5`. Client writes denied (`patientFileReports.jsx:7-11`).

```js
PatientFileReports.demandFormTypes = {
  FIRST: 'first',
  EXTENSION: 'extension'
};
```

`patientFileReports.jsx:13-16`. These two values describe the **kind of demand form** to generate (initial application vs. extension request) — they are *not* a classification of the report row itself. The collection schema does not have a `type` field.

### Schema

`patientFileReports.jsx:18-30`:

| Field | Type | Required | Meaning |
|---|---|---|---|
| `patientFileId` | `String` (Id) | yes | Owning dossier. `patientFileReports.jsx:19` |
| `name` | `String` | yes | The report title shown in the document list and on the editor page. `patientFileReports.jsx:20` |
| `report` | `String` | no | The HTML body produced by the rich-text editor. Initially seeded server-side with `<h1>{practice.name}</h1>` on insert (`methods.jsx:46`). |
| `tags` | `[String]` | no | Free-form labels. Sorted on edit (`methods.jsx:92`), unioned into the dossier's `files.tags` index. `patientFileReports.jsx:22-23` |
| `treatmentId` | `String` (Id) | no | Optional link to a `Treatments` row. When set, the report is shown under that treatment's documents view. `patientFileReports.jsx:24` |
| `createdAt` | `Date` | yes | Stamped by the base Collection on insert. `patientFileReports.jsx:25` |
| `removed` / `removedAt` | Boolean / Date | no | Soft-delete flags. `patientFileReports.jsx:26-27` |

`PatientFileReports.publicFields = { removed: 0, removedAt: 0 }` — everything except the soft-delete flags is published (`patientFileReports.jsx:32-35`).

There is **no `report.type`, `signedAt`, `authorId`, `version`, or `template` field** — reports are stored as a single editable HTML blob with no versioning, no signatures, no audit, no templates.

## Methods

Defined in `app/imports/api/patientFileReports/methods.jsx`. All `PermissionValidatedMethod` with `subscription: true`.

### `patientFile.reports.add`

`methods.jsx:10-54`.

- Inputs: `{ name, patientFileId, tags?, treatmentId? }`.
- Permission: `patientFile.reports.add`.
- Run:
  - Dedupes tags via `_.uniq`.
  - Seeds the report body with `<h1>{practice.name}</h1>`.
  - `$addToSet`s the new tags into the dossier's `files.tags` index.
  - Inserts the row.
- Returns the new `_id`.

### `patientFile.reports.edit`

`methods.jsx:56-100`.

- Inputs: `{ reportId, patientFileId, report: { name, report?, tags?, treatmentId? } }`.
- Permission: `patientFile.reports.edit`.
- Run:
  - Sorts tags via `_.sortBy`.
  - `$set`s the new fields.
  - Calls `PatientFilesUtil.updateTags(patientFileId)` to recompute `files.tags` from all live reports + documents.

The editor page (`PatientFileReportPage.jsx:82-98`) wires this method to the form-group `valueChanges` stream with a 500 ms debounce — every keystroke triggers an update with the full report state, with a `startUpdate` / `endUpdate` saving indicator (`status/saving/SavingStatus`).

### `patientFile.reports.delete`

`methods.jsx:102-132`.

- Inputs: `{ reportId }`.
- Permission: `patientFile.reports.delete`.
- Run:
  - Soft-removes the row.
  - Recomputes the dossier's `files.tags`.

### `patientFile.demandForm.add`

`methods.jsx:134-171`. The PDF generator.

- Inputs: `{ fillInfo, patientFileId, startDate, tags?, treatmentId, type }`. `type` must be `"first"` or `"extension"` (validated on the server side via the form-fill code's switching, see `forms.util.ts`).
- Permission: `patientFile.reports.add`.
- Run: server-side delegate to `PatientFilesUtil.generateDemandForm(patientFileId, this.userId, options)` (`patientFiles/server/util.js:34-106`).

What `generateDemandForm` does:

1. Resolves the form template path based on `type`, `treatment.type`, and the therapist's `locale()`:
   - For extension: `assets/app/files/demandForm/form_extension_<locale>.pdf`
   - For non-standard treatment types: `assets/app/files/demandForm/form_<treatmentType>_<locale>.pdf`
   - Otherwise: `assets/app/files/demandForm/form_standard_<locale>.pdf`
   (`patientFiles/server/util.js:19-32`)
2. Reads the PDF from disk.
3. Calls `fillInForm(pages, ...)` from `patientFiles/server/forms.util.ts` to fill in the patient demographics, treatment metadata, INSZ, RIZIV number, etc. The `fillInfo` boolean controls whether the practice's administrative info is auto-filled.
4. Saves the modified PDF, writes it to the `Documents` collection (which then uploads to S3).
5. Updates the dossier's `files.tags` with the supplied tags.
6. Returns the new `documentId`.

The result is a `Documents` row, not a `patientFileReports` row — meaning generated demand forms appear in the unified Documenten search alongside uploads, but the rich-text editor never opens for them.

## User-visible behaviour

### Adding a report

Three sub-flows from the same `AddReportPopup`:

- **Aanmaken** — typing a name and optionally picking tags. Submitting calls `addPatientFileReport`. On success the user is navigated to `patientfiles.reports.view` where the empty editor opens. Initial body is `<h1>{practiceName}</h1>`.
- **Uploaden** — drag-and-drop or file picker; submitting fires `Documents.insert` with `meta.patientFileId` and `meta.tags` for each file. Each file uploads to S3 via `documentsConfig.onAfterUpload` and the user is navigated to the document viewer if exactly one file was uploaded.
- **Formulieren** — picks a `startDate`, a `first / extension` radio, and an `auto-fill admin info` switch. Calls `addPatientFileDemandForm`. On success navigates to `patientfiles.documents.view` for the generated PDF. Demand-form templates support both NL and FR via separate PDF assets.

### Editing a report

The editor (`PatientFileReportPage.jsx`) shows:

- Name input.
- Tag select (only when `report.treatmentId` is unset — reports linked to a treatment do not allow tag editing).
- Download button — produces a `data:text/html` link via `htmlToDoc()` that wraps the report body in MS-Word-friendly XML, with a `.doc` extension. Clicking it triggers a download.
- Delete button — confirm dialog → `deletePatientFileReport`.
- The `TextEditorLoadable` body — a lazy-loaded rich-text editor.

Auto-save: every keystroke debounces 500 ms, then calls `editPatientFileReport` with the entire form state. Save status is reflected in a global "saving" indicator via `startUpdate` / `endUpdate` in `modules/status/saving/SavingStatus`.

### Demand-form generation

The "Formulieren" tab is conditionally rendered based on `enableDemandForm` (a prop on `AddReportPopup`). When the popup is opened from the dossier-level overview, `enableDemandForm` is false (no treatment context); when opened from a treatment context inside a treatment panel it is true and the form is created against that treatment id.

Form types in i18n:

- `patient.reports.add.demandForm.firstDemand` = "Aanvraagformulier voor terugbetaling" (`client/nl.i18n.js:409`)
- `patient.reports.add.demandForm.extension` = "Kennisgeving van verlenging" (`client/nl.i18n.js:407`)
- `patient.reports.add.demandForm.fillInfo` = "Beroepsgegevens automatisch invullen" (`client/nl.i18n.js:408`)
- `patient.reports.add.demandForm.startDate` = "Datum vanaf" (`client/nl.i18n.js:411`)

## Permissions

- `patientFile.reports.add` — practice owner / admin (`practiceUsers.jsx:49, 114`); per-file `admin` / `default` (`patientFileUsers.jsx:14, 37`).
- `patientFile.reports.edit` — same.
- `patientFile.reports.delete` — same.

The per-file `owner` virtual role (creator-by-metaData) does **not** grant any of these — only `patientFile.view`. So a creator who is a plain "lid" needs the auto-inserted explicit `admin` `PatientFileUsers` row in order to edit reports on their own newly created dossiers (see `patient_creation.md`).

## Notable details

- **Reports are stored as raw HTML, not structured data.** No fields, no schema for headings or sections, no machine-readable assessment scores. The body is whatever the rich-text editor emits.
- **No versioning, no signatures, no immutability.** Every save overwrites the body. There is no signed-at timestamp, no author lock, no co-signature workflow. Two users could in theory race-edit the same report — the last 500 ms-debounced save wins.
- **Initial body is always seeded with the practice name as `<h1>`.** This is a fixed convention from `methods.jsx:46`. It is meant as a header banner; users can delete it.
- **Templates do not exist.** There is no concept of "report template" in the patientFileReports collection. Every new report starts from the seed `<h1>` and that's it.
- **Download exports are MS-Word HTML.** `htmlToDoc()` (`PatientFileReportPage.jsx:38-67`) wraps the body in an `<xml><word:WordDocument>` block and serves it as `data:text/html`. The downloaded file has a `.doc` extension and opens in Word but is technically HTML.
- **Tag editing is hidden for treatment-linked reports.** When `report.treatmentId` is set, the tag select is not rendered (`PatientFileReportPage.jsx:148-159`). The reasoning is not commented but suggests treatment-linked reports inherit their classification from the treatment.
- **Demand forms are PDFs only.** The "Formulieren" tab uses `pdf-lib` to fill in stock templates from `assets/app/files/demandForm/`. Different RIZIV treatment types load different base PDFs (`forms.util.ts` defines `NON_STANDARD_FORM_TREATMENT_TYPES`).
- **Demand-form templates exist per locale.** `_<locale>.pdf` suffix in the file path. Practices that use FR users vs NL users will get different forms.
- **The `tags` array on the row is sorted on edit but not on add.** Add uses `_.uniq`, edit uses `_.sortBy` (`methods.jsx:45, 92`).
- **Files.tags is recomputed from scratch on edit / delete**, not incrementally — `PatientFilesUtil.updateTags` runs a `distinct("meta.tags", { "meta.patientFileId": id })` on Documents and a `distinct("tags", { patientFileId: id })` on PatientFileReports, then `_.union`s them and sets `files.tags` (`patientFiles/server/util.js:128-141`).
- **Reports authored from a treatment go to the same editor as dossier-level reports.** The two routes `patientfiles.reports.view` and `patientfiles.treatments.reports.view` both render `PatientFileReportContainer`, only differing in their breadcrumb parent.
