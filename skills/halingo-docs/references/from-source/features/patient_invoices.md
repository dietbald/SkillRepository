## What it is

The `patientFileInvoices` sub-module is the heart of Halingo's billing surface: every reimbursable event delivered by a logopedist eventually ends up on a *factuur* aimed at the patient (or, under derdebetaler, at the Ziekenfonds with the patient invoice still acting as the source-of-truth for amounts and certificate numbers).

Despite living in a collection literally named `invoices` (`patientFileInvoices.js:11`), this *is* the patient-side invoice. The Stripe SaaS invoices live in `stripeInvoices` — see `./stripe_invoices.md`.

## Data model

### Collection

`PatientFileInvoices = new Collection("invoices")` (`patientFileInvoices.js:11`). Client-side `insert/update/remove` are denied (`patientFileInvoices.js:30-40`); all writes go through Meteor methods.

`PatientFileInvoices.MAX_TREATMENTS_PER_CERTIFICATE = 34` (`patientFileInvoices.js:28`) is the hard cap on the number of *events* that can be folded into a single certificate (and therefore into a single invoice when generated by event ID, since the bulk path slices in chunks of 34: `methods.js:90-101`).

### State enums

```js
// patientFileInvoices.js:13-26
InvoiceStates = {
  OPEN: "open",
  UNPAID: "unpaid",
  PARTIALLY_PAID: "partially_paid",
  PRINTED: "printed",
  MAILED: "mailed",
  PAID: "paid",
};
InvoiceInsuranceStates = {
  OPEN: "open",
  PRINTED: "printed",
  PAID: "paid",
};
```

There is also a `"void"` state, which is allowed by the schema (`patientFileInvoices.js:147-149`) but is not in the `InvoiceStates` enum and is not visibly settable through any method I located. The `InvoiceTemplate1` print template renders a giant red "ONGELDIG" overlay if the invoice state is `"void"` (`InvoiceTemplate1.jsx:151-168`). The `void` filter exists in `PatientFileInvoices.filters.void` (`patientFileInvoices.js:199-202`) and pairs `state: "void"` with `isCanceled: true`. 

> ⚠️ The `void` state appears to be set only by manual database edits or by an admin path I did not locate; the user-facing cancel flow only sets `isCanceled: true` (`methods.js:622-626`).

### Certificate sub-document

Each entry of `certificates` (one per `(treatmentId, bilanId)` pair) has the schema (`patientFileInvoices.js:73-111`):

```js
{
  bilanId,                       // optional — null when treatment has no bilan (e.g. supplementaryInsurance)
  bilanType,                     // initial / evolution / relapse / extension
  certificate: {                 // PatientFileInvoices.certificateSchema
    approvalDate,
    numbers: [{ createdAt, number }],   // ← printed certificate numbers, one entry per print
    prescriberName,
    prescriberRiziv,
    prescriptionDate,
    treatmentEnd,
    treatmentStart,
  },
  doesNotRequireCertificate,    // true for supplementary-insurance treatments without certificate need
  hasNoReimbursableEvents,       // true if every event in this bilan has hasPayBack === false
  insuranceInvoiceId,            // optional — set when this certificate is folded into a Verzamelstaat
  insuranceInvoiceState,         // mirror of the parent InsuranceInvoices.state
  isTreatmentReimbursed,         // false if treatment.approvalState === DECLINED
  state,                         // overrides insuranceInvoiceState; settable via invoices.certificate.edit.state
  treatmentId,
  treatmentName,
  treatmentType,                 // a / b.1 / ... / supplementaryInsurance
}
```

There is a `custom` validator on the `certificates` array (`patientFileInvoices.js:65-71`) that rejects duplicate `(treatmentId, bilanId)` pairs.

### Helpers (`patientFileInvoices.js:214-286`)

| Helper | Returns | Notes |
|---|---|---|
| `getAmount()` | `Number` | `amount + (administrationCost || 0)` |
| `canEdit()` | `Boolean` | `!isCanceled && state === OPEN` |
| `hasThirdPaid()` | `Boolean` | `isThirdPayer && some(events, isReimbursable)` — only true once a reimbursable event has been folded in |
| `getAmountPatient()` | `Number` | Sum of `event.getAmountPatient()` + `administrationCost`. Per-event: `(isReimbursable ? pricePatient : price) + kmCompensation` (`events.jsx:1402-1404`). |
| `getAmountPatientToPay()` | `Number` | `isThirdPayer ? getAmountPatient() : getAmount()` |
| `getAmountInsurance()` | `Number` | `getAmount() - getAmountPatient()` |
| `getCertificate(treatmentId, bilanId)` | `Object` | `_.find(certificates, ...)` |
| `getCertificates()` | `Array` | The full array, or `[]` |
| `getCompanyNumber()` | `String` | For practice-type invoices: `vatInfo.countryCode + vatInfo.vatNumber`; otherwise `companyNumber` |
| `getEventsOfCertificate(treatmentId, bilanId)` | `Array<Event>` | Filters events to the bilan AND `isReimbursable && code` |
| `getInsuranceState()` | `String` or `null` | `null` if no third-paid certs; otherwise `stateInsurance` falls back to *the worst* state across all certificates by enum index |
| `isInvoiceFromPractice()` | `Boolean` | `meta.type === Practices.invoiceTypes.PRACTICE` |
| `getName()` | `String` | Practice's `vatInfo.companyName` / `name` for practice invoices, otherwise `user.name` |
| `getPracticeFullAddress()` | `String` | Single-line `street, postalCode city` |
| `locale()` | `String` | `meta.locale ?? "nl"` — drives the language of the printed PDF |

The `getInsuranceState()` "worst case" computation is non-obvious — it walks `Object.values(InvoiceStates)` (which is `["open","unpaid","partially_paid","printed","mailed","paid"]`), computes the index of each certificate's `state || insuranceInvoiceState || "open"`, and picks the **minimum** index. So `open` outranks `paid`, meaning a single still-open certificate keeps the whole insurance side open (`patientFileInvoices.js:255-269`).

## Methods (Meteor)

All methods live in `app/imports/api/invoices/patientFileInvoices/methods.js` and use `LoggedInValidatedMethod`. Permissions are checked through `PracticeUserUtil.checkUserPermission`.

### `invoices.add.all.therapists` (`createInvoiceByEventIds`, `methods.js:22`)
Bulk-create invoices from a flat list of `eventIds` belonging to one practice. Walks the events, validates that each therapist on the list has a complete profile (address, bankAccount, riziv, name), groups events by `(patientFileId, userId)`, sorts each group by `(treatmentId, start)`, then **slices each group into chunks of `MAX_TREATMENTS_PER_CERTIFICATE = 34`** and emits one invoice per chunk. Returns `{ success, errors: [{ patient, error }] }`.

### `invoices.add.patient` (`createInvoiceForPatient`, `methods.js:116`)
Create one invoice for one patient by aggregating *all* of that patient's uninvoiced events. Internally calls `EventsUtil.getUninvoicedEvents(userId, { patientFileId, practiceId })` (`api/events/util.jsx:70-92`) and passes the result to `InvoiceUtil.createInvoiceFromEvents`. Does **not** chunk by 34 — will throw `invoices.create.tooManyEvents` if the result exceeds the cap.

### `invoices.edit.state` (`setState`, `methods.js:160`)
Updates `state` and/or `stateInsurance`. Sets `isPaid` to `true` only when `state === PAID` *and* the insurance side is also `PAID` (or non-third-payer). Refuses on cancelled invoices.

### `invoices.certificate.edit.state` (`setCertificateState`, `methods.js:222`)
Per-certificate state mutation. Updates `certificates.$.state` via positional `$elemMatch` filter on `(bilanId, treatmentId)`. Recomputes `isPaid` the same way.

### `invoices.edit.administrationCost` (`setAdministrationCost`, `methods.js:292`)
Sets the optional admin-cost surcharge in cents. Refuses on cancelled or paid invoices. The schema `setAdministrationCostSchema` requires `min: 0`.

### `invoices.search` (`searchPatientFileInvoices`, `methods.js:329`)
The CollectionSearch backend. Searches across `patient.name`, `structuredAnnouncement`, `certificates.certificate.numbers.number`, `amount`, `invoiceNumber`. Has custom search adapters for `amount` (strips `,`/`.` and uses `$where` regex) and `invoiceNumber` (uses `$expr / $regexMatch` after `$toString`). If the caller lacks `patientFile.view`, it forces `filter.userId = this.userId` regardless of any provided `userId`.

### `invoices.edit.structuredAnnouncement` (`setStructuredAnnouncement`, `methods.js:432`)
Updates the structured announcement. **Refused unless the invoice is in `OPEN` state** (`methods.js:458-460`).

### `invoices.print` (`printInvoice`, `methods.js:469`)
Sets `state: PRINTED`. Refuses if the invoice is `PAID` or `void` (returns `1` to short-circuit). Note that *the actual printing* happens in the browser before the call: `PatientFileActions.jsx:54-65` calls `Util.printHTMLElement(Util.componentToStaticHtml(InvoiceTemplate*, ...))` and *only then* fires `printInvoice` to update the DB state.

### `invoices.prints` (`printInvoices`, `methods.js:496`)
Returns invoices in a date range for *one* patient and the calling user. Used by the per-dossier bulk print dialog. Does not flip any state — that's the caller's job (one `printInvoice.call` per result, see `PrintPage.jsx:84-87`).

### `invoices.printInvoicesPractice` (`printInvoicesPractice`, `methods.js:525`)
Returns invoices in a date range across the whole practice (or a single user). Used by the practice-wide bulk print dialog on `FinanceInvoicePanel`.

### `invoices.mail` (`mailInvoice`, `methods.js:568`)
Triggers email delivery via `InvoiceUtil.mailInvoice` (`server/util.js:612-699`) and flips state to `MAILED`. Accepts `reminder: Boolean`. The mail flow:

1. Loads patient → reads `contactDetails.email`, throws `invoices.mail.noPatientEmail` if missing.
2. Loads practice → reads `settings.invoices.mail.text`, throws `invoices.mail.noText` if missing.
3. Picks the `from` address: for member-type invoices it's the user's email and `fullName()`; for practice-type invoices it's `practice.contact.email` and `practice.name`.
4. Renders the invoice through `InvoiceTemplates[meta.template ?? 0]` to static HTML.
5. Generates a PDF via `SharedUtil.generatePDF(html)` (`api/shared/server/util.js:8-51`, uses `html-pdf`).
6. Sends via `HalingoEmails.sendEmail` using the chosen `InvoiceMailTemplates[settings.invoices.mail.template ?? 0]`.
7. **Records the dispatch in the `emails` collection** as `{ entityType: "PATIENT", type: "PATIENT_INVOICE" | "PATIENT_INVOICE_REMINDER", status: "UNKNOWN", invoiceId, ... }` (`server/util.js:685-696`).
8. Returns `result.success`. If the underlying email send returned success the method then flips the invoice's `state` to `MAILED` (`methods.js:595-598`).

The `emails` collection is the integrations agent's territory — see `./email_delivery.md` (sibling file, may not exist yet).

### `invoices.cancel` (`cancelInvoice`, `methods.js:602`)
Sets `isCanceled: true`. Refuses if the invoice is paid (`isPaid: true`). **Crucially**, before the update it `Events.update({ invoiceId }, { $unset: { invoiceId: true } }, { multi: true })` — so the underlying events become available for re-invoicing immediately (`methods.js:622`).

### `invoices.certificates.generate` (`generateCertificate`, `methods.js:631`)
Re-runs `_generateCertificates(invoiceId, treatmentIds?)` for one (or all) treatment IDs. The implementation lives in `server/util.js:244-475` — see "Certificate generation" below.

### `invoices.certificates.print` (`printCertificate`, `methods.js:697`)
Records that a certificate was printed. The validation schema is `printCertificateSchema` (`methods.js:671-696`), which accepts:
- `bilanId`, `treatmentId`, `invoiceId` — identifies which certificate inside the invoice.
- `certificateNumber.bookNumber` (regex `\d{2}\*\d{4}`), `certificateNumber.certificateNumber` (1–50), `certificateNumber.printerNumber`.
- `mode` — `manual` or `printer` (`Meteor.users.certificateModes`).
- `offset.{left,top}` — printer-mode margin in cm.
- `therapistInformation`, `therapistInformationPractice` — printer-mode toggles for embedding therapist/practice identification on the form.

The method:
1. Validates that for `manual` mode both `bookNumber` (matching `\d{2}*\d{4}`) and `certificateNumber` are present, otherwise that `printerNumber` is present.
2. Persists the user's settings: `settings.certificates.{mode, offset, therapistInformation, therapistInformationPractice}`, plus `profile.certificateNumber.{bookNumber, certificateNumber}` for manual mode or `profile.certificateNumber.printerNumber` for printer mode.
3. `$addToSet`s a new `{ createdAt: new Date(), number: ... }` entry into `certificates.$.certificate.numbers` — the manual `number` is `bookNumber/certificateNumber`, the printer `number` is the bare `printerNumber`. Multiple printings → multiple entries.

See `./certificate_printing.md` for the matrix-printer alignment story.

## User-visible behaviour

### Generation flow

The full create path lives in `server/util.js:21-242` (`_createInvoiceFromEvents`). High-level:

1. **Caller context check.** Picks `practice.settings.invoices.type` (defaults to `MEMBER`) to decide whether the invoice is sent in the *therapist's* name or in the *practice's* name.
2. **Completeness check.** For member-type, requires therapist's address, bankAccount, riziv (if `user.settings.invoices.rizivRequired` is true — default `true`), names, companyNumber, plus practice contact email. For practice-type, requires practice address, bankAccount, companyNumber-or-vat, contact, plus therapist names and (if required) riziv. Throws `invoices.create.incompletePractice*` / `incompleteUser*`.
3. **Per-event sanity.** All events must share `patientFileId` and `userId` and pass `event.canBeInvoiced()` (`events.jsx:1363-1368` — defined as `patientFileId && !invoiceId`).
4. **INITIAL_BILAN expansion.** For each event with `meta.type === 2` (initial bilan), the original event is **split into N child events of 30 minutes each**, where `N = event.getSessionCount()`, the price is divided by `N`, `kmCompensation` is only kept on the first slice, and the start times are stepped forward by 30 minutes. This ensures the matrix-printer certificate has one row per 30-minute reimbursement unit (`server/util.js:97-110`).
5. **Patient completeness.** Throws `invoices.create.incompletePatientFile` if the dossier is missing address, name, salutation, or — when `isThirdPayer` — `healthInsurance`.
6. **Per-event price + payback computation.** For every event:
   - `event.price = event.price || prices.price` (where `prices = event.getInsurancePrices()`).
   - `event.pricePatient = event.price - payback`. The payback amount is:
     - `treatment.supplementaryInsurance.payback` if the treatment is on supplementary insurance.
     - `prices.payback_with` if the patient has `hasIncreasedReimbursement()` (CG1 odd, "verhoogde tegemoetkoming").
     - `prices.payback` otherwise.
   - **De-conventioned discount:** if the event date is after `2024-04-01`, the user's `profile.isDeconventioned === true`, the treatment is *not* supplementary-insurance, and the patient does *not* have increased reimbursement, then `pricePatient = price - Math.ceil(prices.payback * 0.75)` (`server/util.js:152-158`). I.e. the patient pays 25% of what the convention payback would have been on top of the regular remgeld.
   - `amount += event.price + event.getKmCompensation()`.
7. **Invoice numbering.** `invoiceNumber = practice.invoiceNumber + 1`. The number is per-praktijk and monotonic. There is a `// TODO not ok in case of multiple servers` comment (`server/util.js:164`) — the increment-and-save is **not atomic** and is racey under multi-process load. The new value is written back to `Practices.update(practiceId, { $set: { invoiceNumber } })` after the insert (`server/util.js:230`).
8. **Structured announcement.** Built from `practice.settings.invoices.communicationStructure` (default `NAME-DATE-NUMBER`, `practices.jsx:26-31`). The placeholders (`server/util.js:169-179`):
   - `FULLNAME` — `patientFile.name().toUpperCase()`
   - `DATE` — `moment().format("YYYYMMDD")` (today's date, not the invoice's first event)
   - `MONTH` — `moment(firstEvent.start).format("MMM").replace(".", "").toUpperCase()` (locale-dependent month abbreviation)
   - `NAME` — first 3 letters of `lastName`, uppercase
   - `NUMBER` — invoice number padded to 3 digits with leading zeros
9. **Insert.** Snapshot `address`, `bankAccount`, `companyNumber`, `meta` (the merge of practice/user invoice settings), `patient`, `practice`, `user`, `insuranceAddress`, `insuranceCode`, `insuranceName`, `events`, `amount`, `invoiceNumber`, `structuredAnnouncement`, `isThirdPayer`, `hasIncreasedReimbursement`. Returns `invoiceId`.
10. **Side-effects.** `Practices.update($set: invoiceNumber)`. `Events.update($set: invoiceId)` for every involved event. Then call `_generateCertificates(invoiceId, Object.keys(eventGroupsByTreatment))`.

### Certificate generation

`_generateCertificates(invoiceId, treatmentIds?)` (`server/util.js:244-475`) is called once at invoice creation and again on demand via the `invoices.certificates.generate` method.

Key behaviour:

- If the only event group is "no treatment" (untreated events on a generic invoice), it inserts a single placeholder certificate `{ treatmentId: null, doesNotRequireCertificate: true }` and throws `INVOICE_HAS_NO_EVENTS_WITH_TREATMENT`.
- If no certificate has yet been *printed* (no `certificate.numbers.length`), it refreshes the patient's SSN, the user's `companyNumber`, and the insurance address from the latest patient/user record. This means re-running generate before printing pulls in any post-creation profile fixups.
- For each treatment:
  - Loads the `Treatment` document. If gone (`!treatment`) it marks every event in that group `isReimbursable: false` and skips.
  - `isTreatmentReimbursed = treatment.approvalState !== DECLINED`.
  - `doesNotRequireCertificate = treatment.isSupplementaryInsurance() && !treatment.supplementaryInsurance.isCertificateNeeded`.
  - For each event, calls `treatment.getValidBilan(event.start)` (`treatments.js:243-249` — finds a bilan whose `[start, end]` window contains the event date). Special case: an `EVALUATION_SESSION` (`event.isExamination()`) can fall outside any bilan window if it predates the earliest bilan; the code then assigns it to the earliest bilan retroactively.
  - If still no bilan and the treatment is supplementary-insurance or `meta.type === 6`, a synthetic bilan with `_id: "NO_BILAN"` is used.
  - **Once an event is moved into a bilan that already has `certificate.numbers.length > 0`** (i.e. a printed certificate), it throws `invoices.certificates.generate.movedToPrintedCertificate` — you can't slip new events into a printed certificate.
  - Walks the events in the bilan and, for each, computes `event.code = treatment.getCodeForEvent(eventDoc)` and `event.isReimbursable = !!event.code && hasPayBack && state !== ABSENT`.
  - Builds a per-bilan `certificate` entry with all the prescriber/approval/treatment-window data from the bilan, **only** if the treatment is reimbursed, the bilan is complete, and there are reimbursable events. Otherwise it leaves the certificate present but with errors collected: `TREATMENT_NOT_REIMBURSED`, `BILAN_INCOMPLETE`, `TREATMENT_DOES_NOT_REQUIRE_CERTIFICATE`, `NO_REIMBURSABLE_EVENTS`.
- After processing, removes any leftover certificate entries that no longer have any events on the invoice.
- Persists the result by writing back `_.omit(invoice, "_id")`.

Returns `{ success: [{certificate}], failed: [{certificate, bilan, errors}] }`.

### State machine

The flow of `state` and `stateInsurance` from a user perspective:

```
                              setState
                          ┌─────────────────┐
                          ▼                 │
   (created)─▶ open ─▶ unpaid ─▶ partially_paid ─▶ paid
                  │                                  ▲
                  ├─▶ printed ───────────────────────┤
                  │                                  │
                  └─▶ mailed  ───────────────────────┘
                  
                  (parallel) cancel ─▶ isCanceled = true
                  (db-only)         ─▶ state = "void"
```

Notes:

- The transitions are *not enforced*: `setState` accepts any pair of `(state, stateInsurance)` from the enums regardless of current value, with the only blockers being `isCanceled` and the `state === OPEN` requirement on `setStructuredAnnouncement`.
- `printInvoice` and `mailInvoice` short-circuit when state is already `PAID` and return `1` instead of writing.
- The aggregate `isPaid` flag is derived: `state === PAID && (!isThirdPayer || getInsuranceState() === PAID)` (`methods.js:213-216`). It is the field used for "open vs paid" listings, the open-bills widget's `isPaid: false` selector, and the cancel guard.
- For derdebetaler invoices, `stateInsurance` follows the lifecycle of the parent Verzamelstaat: when the Verzamelstaat is `setState`'d to `printed` or `paid`, the insurance methods do an `arrayFilters` write to **update every `certificate.$[elem].insuranceInvoiceState` on every linked patient invoice** (`insuranceInvoices/methods.js:94-105`). When the Verzamelstaat is cancelled, the same `arrayFilters` write `$unset`s `insuranceInvoiceId` and `insuranceInvoiceState` on every certificate (`insuranceInvoices/methods.js:351-362`).

### Cancellation

- Cancelling a patient invoice is **soft**: `isCanceled: true`, the row stays in the collection.
- The events are **un-linked** so they can be re-invoiced (`methods.js:622`).
- Cancel is refused if `isPaid` is true.
- Cancel is also refused (in the UI layer, not in the method) if any certificate has been folded into a non-cancelled Verzamelstaat: the user gets the `notCancellableInsuranceInvoiceMsg` confirm dialog instead (`PatientFileActions.jsx:144-166`).
- Cancellation does **not** delete the certificates or the printed certificate numbers; it leaves the historical record intact.

### Bulk print and bulk mail

There is no bulk mail. There is bulk print:

- **Per-dossier bulk print**: `printInvoices` method, fronted by the dialog `PrintInvoices` → `PrintPage` (`patientfiles/invoices/PrintPage.jsx:44-101`). Date-range filter, fetches the invoices, renders each via the configured `InvoiceTemplate*`, joins the HTML with `Util.printHTMLElement`, then iterates the list calling `printInvoice.call({ invoiceId })` per row to flip each one to `PRINTED`.
- **Practice-wide bulk print**: `printInvoicesPractice` method, fronted by the same dialog from the `FinanceInvoicePanel`. Same loop. Optionally restricts to a single `userId`.

There is no bulk PDF download.

### Per-praktijk invoice language — verdict

> **Verdict: CONFIRMED.** `practices.jsx:71-75` defines `"settings.invoices.locale"` with `allowedValues: locales` (the global locale list from `i18n/index`) and `defaultValue: locales[0]`. At invoice creation the practice settings are merged into `meta` (`server/util.js:199-202`), and `PatientFileInvoices.helpers.locale()` returns `meta.locale ?? "nl"`. Every PDF render path (`mailInvoice`, browser-print, certificate-print) reads `invoice.locale()` and passes it to `Util.componentToStaticHtml(... locale)` and `setLocale(invoice.locale())`. The structured-announcement also bakes the practice locale in via `setLocale(user.locale())` *before* generating the date placeholder (`server/util.js:163`). 

> ⚠️ Note that there's a subtle inconsistency: the structured announcement's locale comes from `user.locale()` (the **user**'s profile locale), not from `practice.settings.invoices.locale`. Verify with product before treating these as equivalent.

## Notable details

- **Snapshots are deep.** When you create an invoice, every relevant identifier (practice address, user RIZIV, bank account, patient SSN, insurance address) is **snapshot-copied** into the invoice document. Subsequent edits to the patient file or practice **do not** retroactively change historical invoices. The only field that is refreshed is the certificate's prescriber/approval block, which `_generateCertificates` re-pulls from the live `Treatments` collection on every invocation that has not yet been printed.
- **The events array on the invoice is also a snapshot** — including the computed `code`, `isReimbursable`, `pricePatient`. After invoice creation, the underlying `Events.invoiceId` link is what stops the event from being re-invoiced; if a user edits the live event, the invoice does not see the edit.
- **`MAX_TREATMENTS_PER_CERTIFICATE = 34`** — this is the matrix-printer constraint. Each Belgian *getuigschrift voor verstrekte hulp* has 34 lines for individual care provisions.
- **Race condition on invoice numbering.** `practice.invoiceNumber + 1` is computed on the application server, then written back via `Practices.update`. Under multiple Meteor instances this can collide. The TODO comment is at `server/util.js:164`.
- **The `state: "void"` mystery.** The state is allowed by the schema and recognized by the print template, the filters, and `cancelInvoice`'s pair `(state: "void", isCanceled: true)`, but **no method sets it to `void`**. This is a code-derived gap; ask product whether `void` is a manual-DB-edit thing or whether there's an admin path I missed.
- **Reminder mails are tracked separately** in the `emails` collection as `type: "PATIENT_INVOICE_REMINDER"` (`server/util.js:691`).
- **The `getInsuranceState()` falls back to the `state` of each certificate** when computing the worst case, not to `insuranceInvoiceState`. The two coexist because `setCertificateState` writes `state` while the Verzamelstaat sync writes `insuranceInvoiceState`. Reading code prefers `state || insuranceInvoiceState` (`patientFileInvoices.js:262-263`).
- **`hasPayBack` is the gating flag on individual events** — `_generateCertificates` only marks an event reimbursable if `event.hasPayBack && state !== ABSENT && treatment.getCodeForEvent(eventDoc)` is truthy (`server/util.js:408-414`). The `hasPayBack` flag itself is computed by `EventsUtil.canBePaidBack` at event create/update time and tracks max-1-per-day, max-5-per-week, max-2-bilans, age-18 cutoff, and total-sessions-left rules — see `./event_payback.md`.
- **Administration cost** is denominated in cents like everything else, displayed as a separate row on `InvoiceTemplate1` (`InvoiceTemplate1.jsx:196-206`), and added to revenue on the earnings graph at the **invoice creation date**, not at the date of any underlying event (`server/util.js:560-571`).
- **No invoice can be removed.** `PatientFileInvoices.deny({ remove: true })` (`patientFileInvoices.js:38`). The `removed: Boolean` field exists but is set only by migrations.
