# REST endpoints

Reference for every HTTP endpoint exposed outside of the Meteor DDP/method protocol.

The orchestration entry point is `app/imports/startup/server/rest.jsx`, which is three lines:

```js
import "../../api/subscriptions/server/webhooks";
import "../../api/invoices/stripeInvoices/server/rest";
import "../../api/events/server/rest";
```

These three modules together register **four** HTTP endpoints. All registration uses `WebApp.connectHandlers.use(...)` directly (Connect middleware) — Halingo does **not** use `Picker.route(...)`, `JsonRoutes.add(...)`, or Restivus / `RestivusAPI.addRoute(...)`.

## Endpoints table

Each event is rendered with a translated title and description that may include the therapy plan. | `events/server/rest.jsx:15` |

## Authentication mechanisms

Halingo uses three different auth schemes for the REST surface:

1. **Stripe webhook signature** — only used on `/api/webhooks/stripe`. The HMAC is computed by Stripe and verified server-side via the official `stripe.webhooks.constructEvent` helper. The shared secret is stored in `Meteor.settings.stripe.webhook`.

2. **Shared secret header** — `secret: <Meteor.settings.rest.secret>` on the request. Used by both `/api/invoice/stripe/:id/pdf` and `/api/invoice/stripe/:id/state`. Same secret for both endpoints; rotation requires changing both the consumer and the server config in lockstep. Returns `401 Unauthorized` if the header is missing or wrong.

3. **Per-user agenda key** — `:key` URL parameter, looked up against `PracticeUsers.publicAgendaKey` or `PracticeUsers.privateAgendaKey` (Mongo IDs). Generated by the `practice.user.update.publickeys` method (`practiceUsers/methods.jsx:54`). The user must additionally have `AgendaSettings.exportAgenda = true` for the feed to return data; if not, returns `401 Unauthorized`. There is no signature, no expiry, no revocation list — knowing the key is sufficient to read the agenda.

## Notable issues

- **Stripe webhook errors are swallowed** (`webhooks.jsx:31-37`): if signature verification throws, the server still responds `success`. This means a misconfigured webhook secret would not surface as a 4xx in Stripe's webhook dashboard — the events would simply silently never be processed.
- **The shared `rest.secret`** has no rotation mechanism in code; secret changes require an environment redeploy.
- **`/api/agenda/*` ignores `req.params.pid`** at line 16 (`const practiceId = req.params.pid;` is read but the variable is never used because the route only takes `:type/:key`, not `:pid`). The selector matches by agenda key alone — practice ID is not enforced. This means an agenda key generated for practice A could in principle be queried even after the user moved to practice B; but since `practiceUser.practiceId` is read into `selectorEvents` from the matched `practiceUser` row, the events returned will always reflect the *current* practice the key is associated with.
- **iCal feed includes private appointments by default** — only `:type === "public"` excludes them. Any other value (including typos like `private2`, `priv`, etc.) yields the private feed. Authorisation is therefore "default unsafe" if the URL is constructed with an unexpected `:type`.
- **No CORS headers, no rate limiting, no per-IP throttling** are configured on any of the four endpoints — they ride directly on Meteor's `WebApp.connectHandlers` middleware stack.

## Endpoints not exposed

The following are explicitly *not* present in the codebase, despite being typical for a SaaS of this shape:

- No `/api/health` or `/api/ready` liveness probe.
- No `/api/version` or `/api/build` introspection.
- No public REST surface for patient files, treatments, or events — all data access goes through DDP methods.
- No API key management UI or endpoint — secrets are environment config only.
- No OpenAPI / Swagger document.
